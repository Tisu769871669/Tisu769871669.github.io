## 接口列表要变成任务地图

数据接口文档通常很全，但真正使用时不会从头读到尾。更好用的整理方式是按任务分类：基础证券信息、交易日历、公司信息、行情辅助字段、特殊状态、管理层信息。

这样后续接 Agent 或工作流时，可以先判断用户问题属于哪类任务，再去找对应接口，而不是让模型在一长串接口里猜。

## 字段说明比调用示例更耐用

公开笔记里不需要放个人配置，也不需要放真实调用环境。更值得留下的是字段含义、必填参数、返回字段、时间粒度和适用场景。

```text
用户问题
  -> 任务分类
  -> 接口候选
  -> 参数检查
  -> 字段解释
  -> 结构化回答
```

## 权限和频率要提前进入设计

数据类工具接到 Agent 里时，不能只考虑能不能查到。还要考虑接口权限、频率限制、返回为空、日期不交易、字段缺失等情况。工具返回最好带上状态说明，让上层工作流知道该回答、重试还是降级。

这类 API 笔记的目标不是替代官方文档，而是把官方文档变成可调用的任务索引。

## 工具返回要带数据语义

金融数据接口返回的字段很多，Agent 如果只拿到原始表格，很容易解释错。工具层最好把字段含义、时间范围、单位和空值原因一起返回。这样上层回答时可以说明数据来自哪里、代表什么。

对于多个接口组合的问题，工具还要说明拼接逻辑。比如先查交易日，再查对应日期数据，再解释字段变化。链路清楚，结果才不容易被误读。

## 时间问题要特别处理

金融数据有明显的时间约束：交易日、公告日、数据更新时间、历史回溯范围都可能影响答案。用户问“最新”或“最近”时，系统应该先确定可用数据的最新日期，而不是直接使用当前日期。

如果接口返回为空，也要判断是参数错、非交易日、权限不足，还是数据本身暂未更新。空结果不是一种结果，而是一类需要解释的状态。
