## 单脚本会越来越难维护

LeadGen 表面上只是搜索和筛选，实际会牵出很多边界：浏览器采集、结果去重、规则过滤、评分、日报、失败兜底。所有逻辑挤在同一个脚本里时，调一个规则就可能影响整条链路。

拆成 Skill 包以后，目录结构反而比代码聪明。采集脚本只管把候选结果落到稳定格式，过滤脚本只管判断是否值得继续看，后处理脚本只管排序、去重和输出。

```text
collect
  -> filter
  -> postprocess
  -> report
  -> status
```

## 稳定 schema 比一次搜索结果更重要

采集结果最好先落成行式记录，字段保持固定：查询词、标题、链接、摘要、来源、位置、时间。后面所有步骤都只依赖这个 schema，而不是依赖浏览器页面当时长什么样。

过滤阶段不要急着追求复杂模型判断。早期可以先把官方站、招聘页、新闻页、重复域名、不可访问页面分出来。只有格式稳定，后面才方便加更细的规则和人工复核。

## 失败也要产出文件

浏览器自动化经常遇到验证码、跳转、空结果、网络错误。最麻烦的不是失败，而是失败以后什么都没留下。即使没有结果，也应该产出空表、状态文件和简短原因。

第二天回看时，可以区分“确实没有合适结果”和“采集链路断了”。这种小约定，比一次漂亮的演示更能支撑长期使用。

## 评分规则要能解释

LeadGen 的排序不要只给一个总分。更好用的是把分数拆成来源可信度、匹配度、可联系性、重复度和风险信号。这样结果靠前时，人工复核能知道它为什么靠前。

评分规则也要允许保守。宁可少给一些候选，也不要把明显不相关或不可联系的页面混进日报。管道早期最重要的是稳定和可解释，不是一次抓到尽可能多的结果。

## 日报要记录缺口

日报不只展示 Top 结果，还应该记录本轮缺口：哪些查询没有结果，哪些来源被阻断，哪些域名重复过多，哪些页面需要人工确认。缺口记录能指导下一轮改查询词、改过滤规则或补数据源。

自动化管道长期可用，靠的不是单次结果数量，而是每次运行后都能说明哪里还不够。
