## 用户敢不敢用，比模型会不会答更重要

这个 PoC 最明显的感受是：前端做 AI 功能，不能只看能不能返回结果，而要看用户敢不敢用。在 IDE 场景里，选区读取、侧边栏预览、手动确认、结果写回，这条链路比一次请求本身更重要。

模型可以先回答，但代码不能先污染。交互被拆成三段：先读上下文，再生成候选结果，最后由用户确认后写回。这样前端能把生成中、可预览、可撤销、失败可重试这些状态讲清楚，用户也更容易建立信任。

## Adapter 和 RAG 是后端的稳定层

后端把模型调用包进 OpenAI Compatible Adapter，把不同提供方统一成一套接口，避免业务层绑死某一家模型。RAG 负责补专业知识和项目资料，减少通用模型在专业语境里的胡写。

```text
IDE 选区
  -> 前端命令 / 侧边栏
  -> FastAPI 编排层
  -> RAG 检索
  -> 模型 Adapter
  -> 预览结果
  -> 用户确认写回
```

## 先别急着写回

这个项目里最容易忽略的地方是写回。生成结果看起来能用，不代表应该马上覆盖用户代码。前端至少要给出预览、确认和取消。

这篇先记这个结论：AI 功能不只是在页面上多一个回答框，还要让用户知道它做了什么、会改哪里、能不能撤回。

## 前后端契约要比模型更稳定

PoC 阶段模型随时可能换，本地服务也可能从 mock 切到真实调用。前后端契约要先固定：请求里有哪些上下文，响应里有哪些引用、告警、候选结果和错误类型。

前端只要依赖稳定契约，就可以独立打磨交互；后端只要保证契约不破，就可以替换检索策略或模型 Adapter。这样项目不会被某一次模型接入绑死。

## 验收要覆盖闭环

只验证“能生成文字”不够。IDE AI 功能至少要验收选区读取、请求发送、检索引用、模型返回、预览展示、确认写回、取消写回和失败提示。每一环都能跑通，才算最小闭环成立。

测试也可以按这条链路拆：前端状态测试、后端接口测试、RAG 检索测试、写回动作测试。PoC 阶段把这些边界测住，后面继续加功能才不容易散。
