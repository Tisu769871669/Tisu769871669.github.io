## 起点不是通用平台，而是一个具体客服

最早的问题很具体：客服 Agent 要能接住真实用户消息，能按固定语气回复，能查业务数据，能通过 HTTP 入口接进外部系统。

这种阶段通常容易把注意力放在 prompt 上。但真正接入以后，问题会往工程侧移动：请求格式是否稳定、会话标识是否可靠、实时数据是否可查、外部渠道是否能收到返回、服务异常时是否能定位。

```text
具体客服 Agent
  -> HTTP Bridge
  -> 业务接口接入
  -> 线上部署与排障
  -> 并发和会话隔离
  -> Agent Pool 通用层
```

这里的关键变化是：前半段回答“一个客服 Agent 怎么做出来”，后半段回答“多个客服 Agent 怎么稳定跑起来”。

## 五层模型更容易复用

一个可复用的智能客服系统，至少可以拆成五层。

```text
渠道接入层
  -> 会话桥接层
  -> Agent Pool 调度层
  -> 业务能力层
  -> 人格与知识层
```

渠道接入层负责外部系统、聊天平台、反向代理和 HTTPS。会话桥接层负责协议兼容、鉴权、会话标识、FAQ 注入、防抖和 Agent 调用。Agent Pool 调度层负责把外部看到的 logical agent 映射到内部 worker pool。业务能力层负责商品、订单、物流、用户、消息发送等实时接口。人格与知识层负责语气、规则、边界、FAQ 和 workspace 配置。

这个分层的价值很直接：复制到新客服时，不必重写整套链路，只需要替换人格、知识、业务接口和路由配置。

## 具体客服阶段先拆清楚三类资产

客服 Agent 不能只靠一个超长 prompt。公开可复盘的部分，可以拆成三类资产。

第一类是人格资产。它负责客服的身份、语气、开场、边界和不能乱说的规则。真实聊天样本更适合沉淀“怎么承接、怎么短句、怎么像真人客服”，不适合直接承担事实正确性。

第二类是稳定知识。FAQ 适合放账号、流程、固定说明、常见问题等长期稳定的内容。它的职责是减少重复问答，不应该回答强时效问题。

第三类是实时能力。价格、库存、订单、物流、直播状态这类信息必须走业务接口。静态知识库只能兜稳定事实，不能凭记忆猜实时状态。

```text
人格资产：语气、边界、开场、承接方式
稳定知识：FAQ、固定流程、常见说明
实时能力：商品、订单、物流、用户、消息动作
```

这三类资产分开后，Agent 行为会更可控。人格负责“怎么说”，FAQ 负责“稳定事实”，业务接口负责“当前事实”。

## Bridge 是系统边界，不只是转发

HTTP Bridge 不只是把消息转给 Agent。它是外部系统和 Agent 运行时之间的边界。

Bridge 至少要处理这些事情：

- 请求字段兼容；
- 会话标识归一；
- 目标 Agent 路由；
- 鉴权与调用来源记录；
- 同会话串行；
- 碎片消息防抖；
- FAQ 或检索上下文注入；
- OpenClaw 调用；
- 超时、错误和降级返回。

这里有一个容易踩的坑：Bridge 不应该重复注入完整人格。人格应该属于 Agent workspace，Bridge 只注入短上下文、检索命中和运行约束。否则同一套规则会在多个位置重复出现，后续调试时很难判断到底哪一层影响了回复。

## 单 Agent 跑起来以后，并发问题才会出现

单个客服 Agent 在 demo 阶段通常够用。一旦进入真实链路，就会遇到这些运行时问题：

- 多个客户同时发消息时，单个 Agent 容易阻塞；
- 同一客户的连续消息必须按顺序处理；
- 不同客户之间不应该串上下文；
- 外部系统不应该知道内部 worker 编号；
- 多个客服人格需要隔离；
- 更新人格、skills、knowledge 后，多个 worker 需要保持一致；
- 线上排障时，需要看到哪个 worker 忙、队列多深、哪个会话正在等待。

这些问题不是某个客服独有，而是智能客服通用基础设施问题。Agent Pool 的抽象就从这里开始。

## Agent Pool 的核心概念

Agent Pool 可以把外部可见的 logical agent 和内部真实执行的 worker agent 分开。

```text
logical agent：外部可见的客服入口
worker agent：内部真实执行的运行副本
template workspace：标准模板目录
worker workspace：从模板同步出的运行目录
conversation queue：同一会话串行
pool wait queue：worker 全忙时排队
sticky binding：同一会话优先复用上次 worker
bridge-owned history：Bridge 保存最近上下文，支持换 worker
```

请求链路大概是这样：

```text
业务入口
  -> HTTP Chat API
  -> DebounceQueue
  -> ConversationQueue
  -> AgentPool 租用 worker
  -> SessionStore 读取上下文
  -> Retrieval Adapter
  -> Prompt Adapter
  -> OpenClaw Runner
  -> worker agent
  -> 写回上下文
  -> 返回 reply
```

这条链路里，外部系统只认识 logical agent，不需要知道 worker agent。相同 `logicalAgent + conversationId` 必须串行，不同 conversation 可以并发。worker 全忙时进入等待队列，超过等待上限就返回明确的繁忙状态。

## 模板 workspace 比直接改 worker 更稳

多 worker 场景里，最怕每个 worker 都被手动改过，最后谁也说不清哪个版本才是准的。

更稳的方式是维护模板 workspace。每个 logical agent 只有一个标准模板，worker workspace 都从模板同步而来。

```text
template workspace
  -> worker-1 workspace
  -> worker-2 workspace
  -> worker-3 workspace
```

人格、FAQ、skills、prompt 更新时，先改模板，再同步到 worker。同步时保留运行态目录，比如日志、临时文件、会话缓存和依赖目录，避免发布过程覆盖线上状态。

这个规则把“每个 worker 都改一遍”的问题，变成“维护模板，再发布到 worker”。

## 排障要从模型扩展到整条链路

真实接入后，经常会出现一种情况：本地调用能拿到回答，但用户侧没有收到消息。

这时不能只盯模型。排查链路要分层：

```text
OpenClaw 是否能回复
Bridge 是否收到请求
Bridge 是否生成非空 reply
反向代理是否转发正常
外部平台是否接收并回传
防抖队列是否只让最后一条产生回复
```

这种排查方式比“模型为什么不答”更接近生产环境。模型能答，只代表其中一层通了。端到端能回到用户侧，才代表整条链路闭合。

## 可观测性必须跟着调度一起做

Agent Pool 如果只有 `/health`，不够排查真实问题。`/health` 只能说明服务活着，不能说明 worker 是否可用、队列是否堆积、某个 logical agent 是否被打满。

更有用的状态包括：

- logical agent 下有多少 worker；
- busy worker 数量；
- pool wait queue 深度；
- conversation queue 深度；
- 最近一次错误；
- 最近一次运行耗时；
- OpenClaw 执行失败次数；
- 检索适配器失败次数；
- 业务接口成功率和耗时。

最小可用形态可以先提供 `/metrics`、`/admin/pool` 和 CLI 状态命令。线上出现问题时，先看运行态，再看日志。

## Adapter 把客户差异隔离出去

通用 Pool 不应该知道具体业务细节。客户差异可以通过 adapter 接进来。

Prompt Adapter 负责不同客服的 prompt 模板。Retrieval Adapter 负责 FAQ、RAG 或外部知识服务。Business Skill 负责真实业务接口。

```text
Agent Pool Core
  -> 调度
  -> 队列
  -> 会话
  -> worker 运行

Adapter Layer
  -> prompt 模板
  -> FAQ / RAG
  -> 业务接口
```

这样开源或通用部分只保留机制：调度、隔离、队列、历史、观测。具体客户的知识、接口、规则留在业务层。

## 有副作用动作要更保守

智能客服一旦接入发送消息、改订单、写数据这类动作，风险会比纯问答高很多。

这类 action 至少需要：

- 明确的调用条件；
- 幂等标识；
- 操作审计；
- 可关闭的保险丝；
- 高风险动作二次确认；
- 失败时的可追踪编号。

只要动作会影响真实业务，就不能当成普通文本生成处理。Agent 可以参与判断，但最终执行层必须有工程保护。

## 从这次复盘沉淀出的检查表

新客服上线前，可以先过一遍这张表。

| 问题 | 检查点 |
| --- | --- |
| 人格源是否明确 | 最终 prompt 由哪些文件组成，是否可重复生成 |
| 稳定知识是否分层 | FAQ 不回答价格、库存、订单、物流等强时效信息 |
| 实时接口是否验收 | 真实接口路径、参数、错误返回都跑过样例 |
| 会话键是否稳定 | 外部渠道能稳定传入 conversationId |
| 同会话是否串行 | 连续消息不会并发污染上下文 |
| 不同会话是否并发 | 多客户不互相阻塞 |
| worker 是否可观测 | 能看到 busy、queue、recent error |
| 模板是否是唯一源 | worker 不作为维护源 |
| 部署环境是否显式 | systemd 下的 Node、包管理器、CLI 路径都写清楚 |
| 高风险动作是否受控 | 写数据、发消息、改状态都有确认和审计 |

## 后续更值得补的部分

第一步是把生产验收闭环做实。健康检查、真实 chat、多 conversation 并发、同 conversation 连续输入、服务重启、反向代理回传，都应该有固定命令和记录。

第二步是把路由表显式化。入口渠道、客户标识、群或应用标识、logical agent、template workspace、worker pool、prompt/retrieval/business skill 配置，都应该能从一张表里看清楚。

第三步是把监控指标补全。平均耗时不够，最好能看到 P95/P99、队列超时次数、worker 最近错误、业务接口失败率、检索失败率。

第四步是把模板同步做成可回滚流程。同步前有 diff，同步后有 checksum，发布记录里能看到版本和操作者，出问题可以回到上一版模板。

## 最后一层抽象

这次真正沉淀下来的，不是某个客服 prompt 的写法，而是一套智能客服工程方法。

先把客服拆成说话风格、稳定知识、实时业务能力和接入链路；再把线上运行暴露出的并发、隔离、同步和观测问题抽成 Agent Pool；最后让新客服只替换业务资产，复用同一套 bridge、pool、模板同步和运维体系。

遇到新客服项目时，先问四个问题：

- 人格源文件在哪里；
- 稳定知识和实时数据怎么分层；
- 外部渠道如何稳定传入 conversationId；
- 请求应该路由到哪个 logical agent 和 worker pool。

这四个问题答清楚，智能客服才有机会从 demo 变成可运营系统。
