## 为什么需要 Bridge

Agent 做到可用以后，第二个问题马上出现：外部系统怎么接，接进来的消息怎么不串，出错以后怎么查。这条记录主要记这些接线问题。

把 CRM、消息桥接、晨报、看板和 Agent 入口拆成独立服务，再通过 HTTP Bridge 把外部系统和 Agent 连接起来。这样每个服务都能围绕输入输出做健康检查、日志和错误处理。

## conversation_id 是隔离边界

这里最关键的是 `conversation_id`。它既是会话隔离键，也是日志追踪键和记忆绑定键。没有它，多人同时交互时很容易串上下文，轻则回复混乱，重则泄露信息。

```text
外部系统
  -> Agent Bridge
  -> 鉴权与协议转换
  -> conversation_id 绑定
  -> 指定 Agent 服务
  -> 工具 / 知识库 / CRM
```

另一件事是把 Persona 做成结构化配置，而不是散落在代码里。人设、规则、开场白、FAQ 都能版本化，后续调整也不会变成到处改 prompt。

## 从脚本到 Skill 包

LeadGen 方向做过一次类似重构：把原本脚本式的采集、过滤、打分、导出流程整理成 Skill 包，主 Agent 负责调度，子 Agent 负责采集，搜索链路保留回退。

这篇先记到这里。后续继续扩这个项目时，日志、重试和失败样本应该先补，因为这些东西比 demo 截图更容易在真实联调里救命。

## 服务拆分要围绕失败边界

多服务协同不是服务越多越好。拆分的判断标准应该是：失败能不能被隔离，日志能不能单独看，重启会不会影响其他链路。CRM、消息桥接、晨报、看板和 Agent 入口各自承担不同风险，把它们拆开后，某个服务异常时不会把整套链路拖垮。

Bridge 这层尤其适合保持无状态或轻状态。会话历史可以落在独立存储里，Agent 配置可以版本化，Bridge 本身只负责协议转换、路由和统一返回。这样上线新 Agent 或调整 Persona 时，不需要修改外部系统接入方式。

## 降级路径要写进接口语义

外部系统最关心的不是内部用了哪个模型，而是这次调用能不能得到明确结果。接口返回里最好区分成功、可重试失败、不可重试失败和降级回答。调用方拿到状态后，才能决定继续等待、提示用户、转人工或结束流程。

会话隔离也要配合降级。一次运行失败，不应该污染整个会话历史；否则下一轮对话会带着错误上下文继续走，问题会被放大。
