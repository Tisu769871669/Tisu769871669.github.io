## 业务需求本质上是状态机

第一次做 Agent 时，注意力很容易先落在 prompt 写得像不像人。后来发现更难的是状态：用户是谁、现在在哪一步、是否缺参数、是否需要查知识库、要不要调工具、失败时怎么兜底。

Dify 的价值在于让流程可视化，但前提是你先把问题拆对。能确定的路径不应该全交给模型自由发挥，而应该被拆成变量、分支、工具和输出约束。

```text
用户输入
  -> 意图识别
  -> 状态变量检查
  -> 工具 / RAG 调用
  -> 条件分支兜底
  -> 结构化回答
```

## 关键状态不能只放在上下文里

多场景 Agent / Workflow 里会出现客服、下单、咨询、分析等流程。做法不是把所有逻辑塞进模型，而是把关键状态放变量里，把可确定规则放在分支里，把需要外部事实的部分交给知识库和业务 API。

长期记忆也不能理解成无限上下文。它应该只保存真正影响下一步决策的偏好和事实，并且要绑定用户或会话标识，避免串扰和污染。

## 工具调用最需要工程化边界

工具接入最常见的问题不是“模型不会调用”，而是参数 schema 不清、错误响应不可解释、鉴权信息缺失、返回字段太复杂。工具描述要让模型看得懂，错误信息要让流程有机会恢复。

拆流程前先问三个问题：用户当前卡在哪，系统应该调用什么工具，失败时怎么返回。三件事没想清楚，节点配得再漂亮也只是暂时能跑。

## 节点拆分要看可测试性

Workflow 节点不是越细越好。拆得太细，画布会变复杂；拆得太粗，又很难定位错误。比较合适的粒度是：每个节点都有明确输入、明确输出，并且能单独用样例验证。

意图识别、参数补全、知识检索、工具调用、结果格式化最好分开。这样某一步出错时，可以直接拿同一份输入重跑对应节点，而不是把整条工作流从头到尾再试一遍。

## 兜底路径也要算主流程

很多 Agent Demo 只把成功路径画得很顺，失败路径临时补一句道歉。实际使用时，缺参数、查无结果、工具不可用、用户意图模糊都是常态。兜底路径应该和成功路径一样被设计，而不是留到最后补。

一个简单做法是给每个关键节点配一条失败出口：缺什么就追问什么，查不到就说明缺少依据，工具失败就返回可恢复状态。这样工作流即使不能完成任务，也不会表现得像突然断线。
