## Bridge 是系统边界，不只是转发

外部系统接 Agent 时，很容易先把重点放在“能不能问到模型”。但一旦进入真实链路，问题会变成：请求从哪里来、要进哪个 Agent、会话怎么续、失败时怎么告诉调用方。

Bridge 这层最好先保持薄，但边界要清楚。它负责把外部消息变成 Agent 能理解的输入，再把 Agent 返回整理成稳定结构。业务系统不应该知道内部编排细节，Agent 也不应该直接暴露运行过程。

```text
外部系统
  -> Bridge 接口
  -> Agent 路由
  -> 会话上下文
  -> 统一响应
```

## 响应结构要方便排查

一次调用至少应该返回三类信息：可展示的回答、可追踪的请求编号、必要的告警信息。回答给用户看，请求编号给排查用，告警信息说明这次结果有没有降级或兜底。

错误也不要直接透出内部堆栈。可以分成参数错误、鉴权失败、目标不存在、执行超时、内部异常。调用方看到的是可处理的类别，排查时再拿请求编号去看内部日志。

## 会话标识不要和内部运行绑死

外部传入的会话标识代表业务对话，内部运行标识代表某一次 Agent 执行。两者可以关联，但不要混成一个东西。这样外部系统可以持续传同一个会话，内部每次运行仍然可以单独追踪、回放和限时。

Bridge 的价值不在于多开一个入口，而在于让外部系统不用猜内部细节。

## 契约字段宁可少一点

Bridge 的请求体不适合一开始设计得很大。最小字段通常包括目标 Agent、会话标识、用户输入、可选上下文和调用来源。其他字段只有在多个调用方都需要时再加入，否则接口会很快变成无法维护的大对象。

响应也一样，先保证回答、状态、请求编号和告警信息稳定。内部工具结果、模型中间状态、调试信息都可以留在日志系统里，不需要直接暴露给调用方。

## 超时和幂等要早想

Agent 调用经常比普通接口慢，超时策略要提前定。Bridge 可以设置运行上限，超过后返回明确状态，并保留后台排查所需编号。调用方不应该无限等待，也不应该因为重试导致重复执行外部动作。

如果某些工具会产生副作用，调用方需要传入幂等标识。Bridge 在执行前先检查是否已经处理过同一请求，避免网络抖动时重复下单、重复发送或重复写入。
