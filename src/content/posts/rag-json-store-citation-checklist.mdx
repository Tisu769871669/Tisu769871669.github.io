## JSON 知识库适合最小闭环

RAG PoC 早期不一定需要马上接复杂向量库。资料量不大时，用 JSON 文件保存切块、来源、标题和更新时间，反而更容易调试。重点是结构稳定，而不是存储形式看起来高级。

每个 chunk 至少要带来源信息。后面模型回答时，引用可以指回 chunk 的标题或文档位置。没有引用的 RAG 很难验证结果从哪里来，也很难判断模型有没有自己补内容。

```text
文档
  -> 切块
  -> 本地索引
  -> 检索候选
  -> 带引用生成
```

## 检索结果要能被人工看懂

检索层最好返回分数、片段、来源和命中原因。哪怕分数算法很简单，只要能解释“为什么这段被拿出来”，调试就会轻松很多。

早期可以先做关键词重叠，再逐步补向量相似度。复杂度应该跟问题走：如果检索错得很明显，先修切块和字段；如果语义相近但词不一样，再考虑向量。

## 引用是质量检查的一部分

引用不是装饰。它能帮助判断三件事：答案是否来自资料、资料是否过期、检索是否跑偏。前端展示引用时不需要很重，至少要让用户知道这段回答参考了哪些材料。

这个小闭环做好以后，RAG 才从“把文档塞给模型”变成“能检查来源的知识调用”。

## 切块策略要跟文档类型走

不同文档不应该用同一种切块规则。接口文档适合按接口或字段分块，教程类文档适合按步骤分块，FAQ 适合一问一答分块，长报告适合按标题层级分块。

切块时还要保留相邻上下文。块太短会丢语义，块太长会降低召回精度。早期可以先用固定长度加标题层级，等失败样本积累后再调整。

## 小型评测集很有必要

RAG 质量不能只靠感觉。可以整理一组固定问题，覆盖直接事实、跨段落问题、查无结果、过期信息和相近概念。每次改切块、检索或生成策略，都用这组问题跑一遍。

评测集不需要大，但要稳定。它能提醒系统是不是因为一次优化，悄悄弄丢了原本能答对的问题。
