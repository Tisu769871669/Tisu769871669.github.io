## 业务层只关心任务，不关心提供方

PoC 里同时需要 mock 模式和 OpenAI Compatible 模式。mock 用来跑通测试和演示，真实模型用来验证效果。如果两套逻辑散在接口里，后面改提供方会牵动太多文件。

Adapter 的边界应该清楚：接收统一请求，返回统一结果，把提供方差异挡在内部。业务层只表达“解释代码”“生成候选”“做规则检查”，不直接拼接模型请求。

```text
业务任务
  -> AiService
  -> ProviderAdapter
  -> mock / compatible provider
```

## mock 不是临时假数据

mock 模式最有价值的地方是稳定。它应该返回可预测结果，让前端状态、接口契约和测试用例不用依赖外部模型状态。只要 mock 和真实提供方共用同一套返回结构，切换成本就会低很多。

流式输出也适合放在 Adapter 层统一处理。上层只关心“现在是否有增量文本”，不需要关心底层事件格式。

## 错误要翻译成产品能处理的状态

模型调用失败时，不要把底层错误原样丢给前端。Adapter 可以把错误分成连接失败、响应格式异常、内容为空、请求超时等类型，再由业务层决定提示、重试或降级。

这条边界越早收住，后面接本地模型、云端模型或测试替身时越不容易乱。

## 能力矩阵要单独维护

不同模型提供方不一定支持同样能力。普通对话、流式输出、结构化返回、向量化、函数式工具调用、上下文长度都可能不同。Adapter 里最好维护一份能力矩阵，让业务层知道当前能力是否可用。

业务层拿到能力信息后，可以决定隐藏某些按钮、改用降级方案，或者提前提示用户。这样功能开关就不会散落在各个页面和接口里。

## 流式输出要和最终结果分开

流式事件适合给前端即时反馈，但最终结果仍然需要一次完整对象。前端不能只依赖增量文本，因为中途断开、格式不完整或引用还没返回时，页面状态会很难收口。

Adapter 可以同时处理两种输出：增量事件用于展示过程，最终对象用于保存、引用、写回和测试断言。这条分界能让流式体验不影响结果一致性。
