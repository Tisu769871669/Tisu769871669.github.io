## 先定目标：它不是展示柜

这个站点一开始容易被做成作品集展示页，但更合适的方向是个人空间：项目、笔记、奇思妙想都放在同一个可搜索的地方。重点不是证明什么，而是把做项目时留下的判断、坑点和方法整理起来。

因此首页没有做大面积营销式介绍，导航也只保留笔记、项目、关于。笔记列表是主入口，项目页只作为索引。这样打开站点时，第一眼看到的是“这里有记录”，而不是一套过度包装的话术。

## 技术栈为什么这样选

第一版用了 React + Vite + TypeScript + MDX。这个组合适合静态内容站，也方便后面继续加交互。

React 负责组件拆分和页面状态。比如导航、侧边栏、卡片、文章详情、项目详情都可以拆成独立组件；搜索框、主题切换、侧边栏折叠这些交互也能用状态管理起来。

Vite 负责开发体验和打包。它启动快、配置少，适合这种静态站。GitHub Pages 用户主页通常用根路径部署，所以构建配置可以保持简单。

TypeScript 的作用是把内容模型约束住。每篇笔记、每个项目、每个技术空间都有固定字段。字段一旦写错，测试和类型检查会尽早暴露问题。

MDX 适合维护技术笔记。正文可以像 Markdown 一样写，前面用 frontmatter 存标题、日期、摘要、标签、项目和阅读时间。这样内容和页面逻辑分开，后续新增笔记时不需要改 UI 结构。

## 内容模型先做小

站点里最核心的数据模型是 `Post` 和 `Project`。`Post` 负责公开笔记，`Project` 负责项目记录。两者通过 `relatedPost` 或项目名产生关联。

```text
MDX 文件
  -> frontmatter
  -> parsePost
  -> posts 时间线
  -> 列表 / 详情 / 搜索
```

frontmatter 固定为标题、日期、摘要、标签、项目、可见性和阅读时间。这个约束看起来有点硬，但好处是列表页、搜索、技术空间筛选都可以依赖同一套结构。

内容模型没有一开始做评论、访问统计、后端管理和复杂分类。对一个个人笔记站来说，最重要的是能写、能找、能安全公开。其他功能都可以后面再补。

## 路由先用轻量实现

当前路由没有引入 React Router，而是用 `window.location.pathname` 做轻量解析。站点页面很少，路径也固定：`/`、`/blog`、`/blog/:slug`、`/projects`、`/projects/:slug`、`/about`。

这种做法的优点是依赖少，逻辑直接。点击内部链接时用 `history.pushState` 更新路径，再触发页面状态变化。对于第一版静态站已经够用。

后面如果页面越来越多，或者需要嵌套路由、加载状态和更复杂的跳转守卫，再换成成熟路由库也不晚。早期先保持简单，能减少理解成本。

## 全文检索先跑本地

搜索逻辑也没有接后端。它把标题、摘要、项目名、标签和正文拼成检索文本，再按关键词过滤。这个方案很朴素，但适合当前内容量。

```text
输入关键词
  -> 拆成 terms
  -> 拼接 title / summary / tags / body
  -> every(term) 命中
  -> 返回过滤后的 posts
```

本地检索的学习点是：不要一开始就引入复杂搜索服务。内容量不大时，前端直接过滤更快，也更容易维护。等笔记数量上来，再考虑索引、分词、权重排序或离线搜索库。

## 技术空间是侧边栏，不是顶部分类

笔记和项目都按技术空间组织：AI / Agent、前端 / IDE、RAG / API、安全 / 研究、部署 / 网络。这个设计不是为了给内容贴标签，而是为了回看时少翻页。

侧边栏默认可以折叠，展开时采用悬浮玻璃层，不挤压右侧正文。这样页面主体仍然是阅读区域，分类只在需要切换时出现。

这个交互里学到一点：侧边栏不一定要成为页面布局的一部分。对于个人笔记站，分类导航更像工具抽屉，应该随时可用，但不抢正文空间。

## 视觉方向：轻玻璃、纸张卡片和网格底

视觉上没有走纯黑科技风，也没有做成明亮营销页。当前设计更接近“桌面笔记”：浅色模式用柔和背景、网格线和玻璃层，深色模式保留黑色系，但避免所有区域都纯黑。

文章卡片用了类似文件条的结构：顶部显示文件名和阅读时间，下面是日期、标题和摘要。这个形式比普通卡片更像笔记系统，也和 MDX 文件组织方式呼应。

深浅色切换通过 CSS 变量完成。颜色、背景、玻璃边框、按钮和代码块都挂在变量上。切主题时只改 `data-theme`，组件结构不用变。

## 详情页要照顾阅读动线

点进笔记后，顶部有返回按钮、日期、阅读时间、项目名、标题、摘要和标签。返回按钮做成明显的胶囊控件，并且保持在可见位置附近，避免读完一篇还要找导航。

正文渲染走一个简单的 Markdown 解析器：标题、段落、列表和代码块都有基础样式。它不追求完整 MDX 渲染能力，但足够支撑当前笔记。后面如果需要表格、引用、图片和自定义组件，再换成更完整的 MDX 渲染链路。

## 隐私校验是发布前的门槛

这个站点的一个重要约束是：公开内容只讲方法，不放敏感细节。手机号、具体投递信息、真实配置、凭据、内部地址和原始业务数据都不应该出现在公开页面。

因此项目里加了隐私扫描和内容测试。测试会检查公开内容里是否出现不该出现的名字、第一人称包装、敏感联系方式和高风险词。这样每次新增笔记后，都要先跑检查再构建。

这类测试不只是防泄漏，也会反过来约束写作风格。笔记会更像复盘和方法总结，而不是个人包装材料。

## 这套栈的学习顺序

如果从零学这套技术栈，可以按这个顺序来：

1. 先学 Vite 项目结构：入口 HTML、`src/main.tsx`、开发服务器、构建输出。
2. 再学 React 组件：把页面拆成 Header、Sidebar、Card、Detail、Footer。
3. 接着学 TypeScript 类型：给文章、项目、分类写清楚数据结构。
4. 然后学 MDX 内容组织：frontmatter 管元信息，正文管记录。
5. 最后学工程校验：测试内容模型、路由、隐私扫描和构建流程。

这个顺序比一上来学复杂框架更稳。先把静态站跑起来，再一点点加交互、类型和测试。

## 后续可以继续补什么

后面比较自然的迭代有几条：

- 给笔记详情加目录和标题锚点，长文会更好读。
- 给搜索加权重，让标题和标签命中排在前面。
- 给技术空间加统计页，看不同方向的笔记密度。
- 给 MDX 渲染加表格、引用和图片能力。
- 给发布流程加 GitHub Actions，构建、测试和隐私扫描都自动跑。

这篇先记到这里。站点本身也是一个项目，设计和技术栈学习都可以继续写进笔记里。
