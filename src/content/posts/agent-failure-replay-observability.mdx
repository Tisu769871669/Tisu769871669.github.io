## 失败样本是下一轮迭代的入口

Agent 出错常见有几类：消息格式不对、会话上下文串了、工具参数缺失、外部服务超时、模型输出不符合约束。只记一个错误文本，很难判断是哪一层出了问题。

每次调用都应该留下最小可回放信息：请求编号、入口、目标 Agent、会话标识、裁剪后的输入、工具调用状态、最终响应类型。公开笔记里不放原始内容，本地日志也要做敏感字段裁剪。

## 先分类，再修 prompt

很多失败看起来像 prompt 没写好，其实是工程边界没定住。比如工具返回字段不稳定、上下文窗口塞了过期消息、错误响应没有结构化，都会让模型表现像“变笨了”。

排查顺序可以固定下来：

```text
入口参数
  -> 会话历史
  -> 工具输入
  -> 工具输出
  -> 模型响应
  -> 兜底路径
```

## 回放不等于重跑线上数据

回放应该使用脱敏后的样本，只保留结构和问题形态。这样既能复现链路，又不会把真实用户输入、业务字段或鉴权信息带到调试页面里。

这类机制前期不需要复杂。能按请求编号找到一次调用的关键节点，就已经比只看终端日志强很多。

## 可回放字段要少而稳定

回放字段不需要把所有内容都存下来。更合适的是保存结构化摘要：入口类型、目标 Agent、会话键、使用的工具、工具返回状态、最终响应状态、耗时和异常类别。原始输入只保留脱敏片段或摘要，避免把调试系统变成新的敏感数据仓库。

如果字段过多，日志会变成另一个难维护的数据模型；字段过少，又复现不了问题。早期可以先固定一套最小字段，再根据真实失败样本补充。

## 失败分类要服务修复动作

失败分类不是为了统计好看，而是为了让下一步动作明确。参数类失败要改契约，工具类失败要改适配层，模型输出类失败要改约束和解析，权限类失败要改运行环境或提示，超时类失败要加队列、重试或降级。

分类一旦和修复动作绑定，复盘就不会停在“又失败了”这种模糊描述上。每条失败样本都能推动一个具体改动。
