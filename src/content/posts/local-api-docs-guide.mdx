## 文档要让别人少猜一步

一份 API 文档如果只能列接口，通常还需要继续补。至少要让另一个人不用看源码，也能把服务跑起来、调通、看懂失败原因。

最小可用的结构通常是：服务信息、配置方式、接口列表、请求和响应示例、错误码，以及一份验收清单。只要这几块齐了，大多数接手的人就不会卡住。

## 边界比完整更重要

更重要的是文档里有没有把边界说清楚。比如这个服务依赖什么后端，输入文件或参数有哪些必填项，输出落到哪里，失败时返回什么状态，健康检查怎么做。

对本地服务来说，示例最好用最小环境就能跑通，所有私有地址、密钥和内部模型配置都应该用占位符替代。

## API 文档也是协作接口

API 文档不用一开始就追求完整，但不要误导人。前端、后端、测试和 AI 工具如果都围绕同一套契约协作，后面联调会少很多猜测。

## 示例要覆盖成功和失败

只写成功示例会让文档看起来很顺，但真正联调时更需要失败示例。参数缺失、格式错误、服务未就绪、下游不可用、结果为空，这些状态都应该给出最小响应结构。

错误响应里要有可读信息，也要有机器能判断的状态字段。前端可以根据状态决定展示提示、允许重试还是引导用户修改输入；测试也可以直接断言错误类型，而不是只看文本。

## 验收清单比长文更实用

本地 API 文档最后可以放一份短清单：服务能启动，健康检查正常，核心接口可调用，错误示例可复现，日志能定位到一次请求，敏感配置没有写进文档。

这份清单看起来普通，但它能让接手者快速判断当前服务是“能跑”，还是“能被别人接起来”。
