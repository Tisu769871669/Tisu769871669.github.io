## 先分清每一层职责

把本地 Web 服务暴露到公网，最容易踩坑的地方不是 FRP 本身，而是没有先分清谁负责传输、谁负责业务、谁负责入口。

比较稳的做法是：本地机器只提供服务，FRP 负责打通隧道，云服务器只做转发和入口控制。如果还要域名和 HTTPS，再让 Nginx 站在最前面。这样一层一层拆开以后，问题会少很多。

```text
Browser
  -> Nginx / HTTPS
  -> FRP tunnel
  -> local web service
```

## 固定排查点

后来固定下来的排查点有几个：端口是否冲突，防火墙是否放行，服务有没有开机自启，本地程序是不是只监听在 `127.0.0.1`。

只要其中一环写得含糊，后面就会出现本地能打开、外网打不开这种老问题。远程访问方案能不能用，不在于配置多复杂，而在于启动链路是否足够简单，出问题时能不能一眼定位到哪一层断了。

## 好方案应该可重建

真正好用的方案，应该是几分钟内能重新搭起来的，而不是只有第一次配置时看起来很厉害。公开笔记里也应该只保留通用拓扑和排查方法，真实域名、地址、端口组合和凭据都不应该出现。

## 安全边界要放在公网入口

本地服务暴露到公网以后，真正需要谨慎的是入口控制。FRP 只解决连通性，不等于解决访问控制。公网入口最好放在反向代理层，统一处理 HTTPS、访问日志、限流和基本鉴权。

本地服务尽量不要直接信任外部请求。即使只是临时演示，也要确认接口是否有危险操作、是否会写文件、是否暴露调试页面。能放内网的管理入口不要放公网。

## 排查顺序从外到内

外网访问失败时，可以按外到内排查：域名解析、云服务器端口、反向代理配置、FRP 服务端、FRP 客户端、本地服务监听。每一层只验证一个事实，避免同时改多处配置。

如果本地能访问、云服务器不能访问，大概率是 FRP 或防火墙问题；如果云服务器能访问、域名不能访问，优先看 DNS 和反向代理；如果偶发断开，再看自启动、保活和日志轮转。
