## 先看请求路径

很多 Web 问题表面上像应用 bug，实际上是请求路径、代理链路或者可见性出了问题。整理调试笔记时，可以先问三个问题：请求有没有出去，响应在哪一层被拦住了，传输过程中有没有被代理改写。

按这个顺序查，通常比先猜漏洞类型更有效。

## 抓包是为了建立可观察性

抓包和代理工具主要是为了看清请求。能看见原始请求、重放请求、TLS 处理、重定向和跨域行为，才比较容易判断问题是在客户端、代理层，还是服务端。

```text
client
  -> local proxy
  -> reverse proxy
  -> application service
  -> downstream dependency
```

## 公开笔记保留方法，不保留攻击细节

公开笔记里不放攻击载荷或利用步骤，只保留调试路径和现象描述。这样既能复用方法，也不会把笔记写成攻击手册。对于真实项目来说，能稳定复现问题，远比知道一个技巧更重要。

## 证据要按链路保存

Web 调试时，截图、请求、响应、代理日志和服务日志最好按同一次问题归档。只保存最终结论，后面很难复查；只保存原始日志，又很难快速看懂。

可以按时间线记录：触发动作、浏览器请求、代理层变化、服务端响应、最终现象。这样后面看笔记时，不需要重新猜当时发生了什么。

## 防御视角更适合公开

安全学习笔记公开时，适合保留防御和排查方法：如何确认请求路径，如何判断代理是否改写，如何定位跨域或重定向问题，如何根据日志缩小范围。具体利用步骤和可操作载荷不应该放到公开页面。

这种写法不会降低笔记价值。工程现场里，能把问题定位到哪一层，通常比记住某个技巧更重要。
