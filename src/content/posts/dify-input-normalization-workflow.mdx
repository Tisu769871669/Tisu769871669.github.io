## 不同输入不要直接进主流程

Agent 工作流容易被输入形态拖乱。文本、语音转写、文档片段、表格摘要看起来都能塞进模型，但它们需要的预处理不一样。主流程如果直接接所有输入，节点很快会堆满分支。

前面加一层 Normalize 会省很多后续分支：把输入统一成文本块、来源、时间、用户意图、上下文长度等字段。主流程只消费统一结构，不关心输入最初来自哪里。

```text
原始输入
  -> Normalize
  -> Chunk
  -> Metadata
  -> Agent Workflow
```

## chunk 要服务后续任务

切块不是越细越好，也不是越长越省事。它要服务后续动作：如果后面要检索，就要保留标题和来源；如果后面要总结，就要保留段落关系；如果后面要执行工具，就要提前抽出关键字段。

归一层还可以做一些轻量判断：输入太短时直接进入对话，输入太长时先摘要，文件类输入先抽结构，再交给后续节点。

## 主流程应该少知道输入细节

工作流的可维护性来自边界。Normalize 做输入处理，Workflow 做业务决策，工具节点做外部调用，输出节点做格式约束。每层职责清楚，后面加新输入类型才不用推倒重来。

多一层不是绕路，是把后面的分支提前收掉。

## 归一层可以顺手做质量标记

Normalize 不只是格式转换，还可以给输入打质量标记。比如是否为空、是否过长、是否疑似重复、是否缺少上下文、是否需要先转写或抽取表格。主流程拿到这些标记后，可以更早做分流。

质量标记不需要复杂，先用几个布尔字段或枚举就够。关键是让后续节点知道输入状态，而不是每个节点都重复判断一遍。

## 元信息要跟着 chunk 走

文档类输入被切块以后，原始来源、标题、页码、段落序号和更新时间都要跟着 chunk 走。后续生成回答、引用来源、排查错引时，都依赖这些元信息。

如果元信息在切块时丢了，后面即使用了 RAG，也只能得到一堆看起来相关的文本，无法确认来源是否可靠。
