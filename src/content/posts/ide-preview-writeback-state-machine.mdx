## 写回前先形成 pending 状态

IDE 场景里，AI 结果一旦直接写回文件，用户就要承担额外确认成本。比较安全的交互是先把结果放进待确认状态，展示变更说明、候选内容和来源，再由用户决定是否写回。

这个状态可以叫 pending write。它不是最终结果，而是一份候选修改。用户可以确认、取消，也可以重新生成。

```text
idle
  -> reading selection
  -> generating
  -> pending write
  -> committed / cancelled / failed
```

## 前端状态要覆盖失败路径

只处理 happy path 会让界面很快失控。生成中、后端失败、检索为空、模型返回空内容、写回失败，这些都应该落到明确状态里。否则用户看到的只是按钮卡住或内容突然消失。

侧边栏可以保存上一次请求编号、当前状态、告警信息、引用来源和待写回内容。这样即使结果不理想，也知道问题发生在哪一步。

## 预览是信任感的一部分

AI 编程工具要让人敢用，不能只靠模型质量。预览、确认和撤销是前端给用户的安全感。尤其是生成结构化文本或代码时，写回动作必须显式发生。

这条规则很朴素：模型可以快一点，文件修改要慢半拍。

## 状态机要有可恢复出口

每个状态都要知道失败后回到哪里。生成失败可以回到 idle 并保留错误信息，写回失败可以停在 pending write，方便用户重新尝试或手动复制。不能让失败直接清空候选结果，否则用户会失去判断依据。

状态机还可以记录上一轮请求编号。前端报错时把编号展示出来，后端日志就能对应到同一次生成，这对调试流式输出和写回异常很有帮助。

## 预览要表达差异，而不是只放全文

如果生成结果很长，只把完整候选内容丢到面板里并不够。更好的是显示改动摘要、影响文件、是否会覆盖选区、引用来源和潜在风险。用户先看摘要，再决定是否展开全文。

这套交互不是为了增加确认步骤，而是为了降低误操作成本。确认动作越明确，AI 写回越容易被长期使用。
